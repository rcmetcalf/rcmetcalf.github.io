     Extenisble markup language, or XML, is the framework used to create all other types of coding languages, like hyper text markup language (HTML) and cascading style sheets (CSS). In the photo above, I used HTML to create a structure for this webpage, but I also used XML. The doctype declaration at the top of the image is XML, as well as the title and links. The doctype declaration tells the computer that this file should be read and rendered as an HTML file. The title defines the webpage's name, and the links are associated with the CSS document and imported fonts. This text is not seen on the page, but the computer uses this metadata to build the page correctly.


Notice the doctype comment at the top.

     XML, HTML, and CSS all have specific standards of use on the web. As Dr. Berquist said, it is helpful to know the web standards "so that we can know when and why we might have to deviate from them." Web standards apply to both the structural layer (HTML and XML) and the presentational layer (CSS). For example, using HTML code for presentational purposes such as background color or font color, although it might show up on the webpage, would not be an appropriate use of HTML. To change presentational features, you should use CSS.

     Besides coding languages, I also learned about relational databases. A single-table database is known as a flat database. It's simple to create and use, but it cannot work for large scale databases because flat databases can produce a lot of redundancy of information. For example, the table below is an example of a flat database:

     Although this table is very small, it has many redundancies. Penguin publishing group is repeated 4 different times under entity "Publisher," and the Penguin email is repeated another 4 times, too. John Steinbeck is listed three times for three separate books. JK Rowling is also listed twice. The table is not sorted alphabetically, and there is no individual specific identifier, like a primary key, to distinguish between authors or titles. In all, this table could be better represented using a relational database.

     The relational database is created by breaking apart the flat database into multiple tables, or entities, with specific and relavent attributes. Using the example above, I would create entities like

* Book -- attributes could be "Title" and "ISBN"
* Publisher -- attributes could be "Name," "City," and "Email"
* Author -- attributes could be "Name," "Age," and "Hometown"

     I'll also need to create a primary key as a unique identifier for each book. I can use the ISBN number. These new categories will be related to each other in the database. Now, when someone wants to find out what books Penguin has published, they can create a query and search the database.
